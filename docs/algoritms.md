# Алгоритмы анализа матчей

Этот документ нужен для ИИ, чтобы он понимал:

- как сейчас реально анализируются матчи в проекте;
- какие данные использует каждый алгоритм;
- какие решения принимаются правилами, а какие через AI;
- что именно можно улучшать в новой стратегии.

Ниже описана фактическая логика по текущему коду, а не желаемая.

## 1. Общая схема анализа матча

Матч проходит несколько слоев анализа:

1. `parser.php` сохраняет prematch-данные матча в `matches`.
2. `stat.php` догружает исторические и табличные метрики по `sgi`.
3. `live.php` догружает live-статистику, счет, время и статус.
4. `Scanner` прогоняет матч через алгоритмы 1, 2 и 3.
5. Если есть сигнал, он публикуется в Telegram и сохраняется в `bet_messages`.
6. По сигналу пользователь может запустить AI-анализ матча.
7. `BetChecker` позже проверяет исход ставки.

То есть текущая архитектура состоит из:

- rule-based scanner;
- AI-слоя поверх scanner;
- post-checker, который определяет win/loss по опубликованным ставкам.

## 2. Общий вход для scanner

Все алгоритмы scanner работают на основе `backend/scanner/DataExtractor.php`.

### 2.1. Какие матчи вообще попадают в анализ

Scanner берет матчи из `matches`, где:

- `time` не пустое;
- `match_status` не пустой.

Дополнительно в `Scanner.php` матч отбрасывается, если:

- не удалось распарсить минуту;
- минута больше `45`, и статус матча не `Перерыв`.

Это значит:

- алгоритмы 1 и 2 работают только в окне первого тайма;
- алгоритм 3 живет в том же потоке, но фактически публикуется только в перерыве.

## 3. Алгоритм 1

### 3.1. Назначение

Алгоритм 1 ищет ставку на гол в первом тайме при счете `0:0`.

Это вероятностная модель. Она использует:

- форму команд;
- H2H;
- текущую live-активность.

### 3.2. Какие поля использует

Из `matches`:

- `ht_match_goals_1`
- `ht_match_goals_2`
- `h2h_ht_match_goals_1`
- `h2h_ht_match_goals_2`
- `time`
- `match_status`
- `live_ht_hscore`
- `live_ht_ascore`
- `live_hscore`
- `live_ascore`
- `live_shots_on_target_home`
- `live_shots_on_target_away`
- `live_shots_off_target_home`
- `live_shots_off_target_away`
- `live_danger_att_home`
- `live_danger_att_away`
- `live_corner_home`
- `live_corner_away`
- `live_xg_home`
- `live_xg_away`
- `live_yellow_cards_home`
- `live_yellow_cards_away`
- `live_trend_shots_total_delta`
- `live_trend_shots_on_target_delta`
- `live_trend_danger_attacks_delta`
- `live_trend_xg_delta`
- `live_trend_window_seconds`
- `live_trend_has_data`

Дополнительно алгоритм 1 теперь использует side-based live-агрегаты из `DataExtractor`, а не только сумму по матчу:

- давление хозяев и гостей отдельно;
- off-target shots по сторонам;
- optional `xG`;
- optional trend-метрики за окно до 5 минут.

### 3.3. Формула

Алгоритм 1 считает:

- `form_score = ((home_goals / 5) + (away_goals / 5)) / 2`
- `h2h_score = (h2h_home_goals + h2h_away_goals) / 10`

`live_score` теперь считается как плавный нормализованный score от `0.0` до `1.0`, а не как 4 фиксированные ступени.

Базовые компоненты live-score:

- общий темп матча по `shots_total`;
- качество моментов по `shots_on_target`;
- интенсивность по `dangerous_attacks`;
- dominance score, который оценивает, насколько давление сосредоточено на одной стороне, а не размазано по обеим командам.

Опциональные компоненты, которые участвуют только если данные есть:

- суммарный `xG`;
- дисциплина по желтым карточкам;
- trend score за последние минуты:
  - `trend_shots_total_delta`
  - `trend_shots_on_target_delta`
  - `trend_dangerous_attacks_delta`
  - `trend_xg_delta`

Важно:

- если `xG` нет, матч не отбрасывается;
- если trend-данных еще нет, trend score просто не участвует в формуле;
- итоговый live score пересчитывается только по доступным компонентам.

Итог:

- `probability = form_score * 0.35 + h2h_score * 0.15 + live_score * 0.50`

### 3.4. Условия сигнала

Ставка разрешается только если одновременно:

- есть данные по форме;
- есть данные по H2H;
- счет первого тайма `0:0`;
- минута от `15` до `30`;
- есть хотя бы 1 удар в створ;
- опасных атак не меньше `20`;
- итоговая вероятность не ниже `0.65` по умолчанию.

### 3.5. Что возвращает

Алгоритм 1 возвращает:

- `probability`
- `form_score`
- `h2h_score`
- `live_score`
- `decision.bet`
- `decision.reason`

### 3.6. Сильные стороны

- логика все еще прозрачна, но стала богаче по live-картине;
- использует одновременно history, текущий live-срез и короткую live-динамику;
- умеет деградировать по неполным данным, не ломаясь из-за отсутствия `xG`;
- учитывает не только общий объем активности, но и то, какая сторона реально давит.

### 3.7. Слабые места

- `form_score` и `h2h_score` все еще довольно грубые и строятся на count-based HT-метриках;
- веса `0.35 / 0.15 / 0.50` выбраны эвристически, а не обучены на `won/lost`;
- trend-модель пока короткая и простая: она использует только дельты за окно snapshot-истории, без более сложной temporal-модели;
- если live feed редко обновляется, trend score будет слабее и менее информативен;
- по-прежнему нет нормальной калибровки вероятности на исторических исходах;
- качество соперников, сила лиги и prematch-контекст пока почти не участвуют в алгоритме 1.

### 3.8. Что улучшать

- откалибровать веса и thresholds по истории `bet_messages.bet_status`, а не держать их эвристическими;
- добавить более надежную temporal-модель:
  - не только baseline vs current snapshot;
  - но и несколько точек внутри окна, чтобы учитывать ускорение/замедление давления;
- добавить side-based prematch-контекст:
  - фаворит/андердог;
  - тотальная линия;
  - сила атаки/обороны из prematch-рынков;
- использовать качество соперников и силу турнира, а не только HT-form/H2H;
- разделить сценарии по типу матча, лиге, фавориту, тоталу и стадии тайма;
- обучить или хотя бы offline-оценить ranking-модель поверх текущих feature'ов вместо чисто ручной композиции.

## 4. Алгоритм 2

### 4.1. Назначение

Алгоритм 2 ищет сценарий:

- фаворит-хозяева;
- счет `0:0`;
- до перерыва хозяева должны забить.

Это не вероятностная модель, а жесткий rule-based filter.

### 4.2. Какие поля использует

Из `matches`:

- `home_cf`
- `total_line`
- `total_line_tb`
- `ht_match_goals_1`
- `sgi_json`
- `time`
- `match_status`
- `live_ht_hscore`
- `live_ht_ascore`

Из `sgi_json` дополнительно берется:

- последние 5 H2H;
- был ли в них гол в первом тайме.

### 4.3. Как строятся данные

Алгоритм 2 формирует:

- `home_win_odd`
- `over_25_odd`
- `total_line`
- `over_25_odd_check_skipped`
- `home_first_half_goals_in_last_5`
- `h2h_first_half_goals_in_last_5`
- `has_data`

### 4.4. Условия сигнала

Ставка разрешается только если одновременно:

- счет первого тайма `0:0`;
- минута от `15` до `30`;
- `home_win_odd <= 1.5`;
- если `total_line = 2.5`, то `over_25_odd <= 1.5`;
- если `total_line > 2.5`, отдельная проверка `ТБ 2.5` пропускается;
- хозяева забивали в 1-м тайме минимум в `3` из `5` последних матчей;
- в H2H был гол в 1-м тайме минимум в `3` из `5`.

### 4.5. Что возвращает

Алгоритм 2 возвращает:

- `decision.bet`
- `decision.reason`
- `algorithm_data` с набором rule-based признаков

Вероятность здесь не считается.

### 4.6. Сильные стороны

- очень понятная бизнес-логика;
- быстро работает;
- легко проверять руками.

### 4.7. Слабые места

- live-картина почти не участвует в решении;
- вся логика держится на жестких порогах;
- пороги `1.5`, `3 из 5`, `15-30 минут` никак не адаптируются;
- не учитывается, кто реально создает моменты;
- H2H используется очень грубо;
- игнорируется качество текущего соперника и контекст лиги.

### 4.8. Что улучшать

- добавить live-подтверждение силы хозяев;
- перейти от бинарных фильтров к scoring-модели;
- тестировать другие пороги по исторической доходности;
- вместо одного market check использовать больше prematch-рынков;
- учитывать `xG`, давление хозяев, разницу атак по сторонам.

## 5. Алгоритм 3

### 5.1. Назначение

Алгоритм 3 ищет ставку:

- индивидуальный тотал выбранной команды больше `0.5`.

Это алгоритм на основе табличных метрик. Он работает не по первому тайму, а по сценарию перерыва.

### 5.2. Какие поля использует

Из `matches`:

- `table_games_1`
- `table_goals_1`
- `table_missed_1`
- `table_games_2`
- `table_goals_2`
- `table_missed_2`
- `live_hscore`
- `live_ascore`
- `match_status`
- `home`
- `away`

### 5.3. Базовая валидация

Алгоритм 3 сразу отбрасывает матч, если:

- нет полного набора табличных данных;
- `table_games_1 <= 10`;
- `table_games_2 <= 10`.

### 5.4. Формулы

Считает 4 коэффициента:

- `home_attack_ratio = (table_goals_1 / 2) / table_games_1`
- `away_defense_ratio = (table_missed_2 / 2) / table_games_2`
- `away_attack_ratio = (table_goals_2 / 2) / table_games_2`
- `home_defense_ratio = (table_missed_1 / 2) / table_games_1`

Порог:

- `> 1.5`

Кандидаты:

- home candidate:
  - `home_attack_ratio > 1.5`
  - `away_defense_ratio > 1.5`
- away candidate:
  - `away_attack_ratio > 1.5`
  - `home_defense_ratio > 1.5`

Если проходят обе стороны, выбирается команда с большей суммой:

- `attack ratio + opponent defense ratio`

### 5.5. Дополнительные условия сигнала

После выбора команды ставка публикуется только если:

- статус матча `Перерыв`;
- выбранная команда еще не забила.

Тогда генерируется рынок:

- `ИТБ <команда> больше 0.5`

### 5.6. Что возвращает

Алгоритм 3 возвращает:

- `decision.bet`
- `decision.reason`
- `selected_team_side`
- `selected_team_name`
- `selected_team_goals_current`
- `selected_team_target_bet`
- `triggered_rule`
- `triggered_rule_label`
- все 4 ratio
- табличные данные команды

### 5.7. Сильные стороны

- отдельный сценарий, не дублирует алгоритмы 1 и 2;
- прозрачен;
- уже умеет выбирать сторону, а не только общий гол.

### 5.8. Слабые места

- используется только таблица, без реальной live-картины;
- формула `(goals / 2) / games` очень грубая;
- не учитывается сила турнира, позиция команды, домашний/гостевой контекст;
- не используется `table_avg` и вообще общий темп турнира;
- порог `1.5` жестко зашит;
- матч анализируется только на статусе `Перерыв`, но без контекста качества первого тайма.

### 5.9. Что улучшать

- объединить табличную модель с live-показателями первого тайма;
- учитывать home/away splits, если они доступны;
- добавить поправку на силу соперника;
- использовать динамический threshold;
- обучить выбор стороны и рынка на истории win/loss.

## 6. AI-анализ поверх алгоритмов

После scanner матч может пройти еще один слой анализа через `GeminiMatchAnalyzer`.

Это не создает самостоятельный сигнал, а интерпретирует уже найденный или выбранный матч.

### 6.1. Какой AI-шаблон используется

AI-шаблон зависит от `algorithm_id`:

- для `algorithm_id = 1` используется prompt для вероятностной ставки на гол в 1-м тайме;
- для `algorithm_id = 2` используется prompt для сценария "фаворит-хозяева забьют до перерыва";
- для `algorithm_id = 3` используется prompt для ставки на `ИТБ выбранной команды больше 0.5`.

### 6.2. Что получает AI

AI получает:

- данные матча;
- live-метрики;
- prematch odds;
- historical HT;
- H2H;
- table stats;
- `sgi_json`;
- исходный текст сигнала;
- `algorithm_payload_json`;
- scanner-вывод:
  - `scanner_probability`
  - `scanner_form_score`
  - `scanner_h2h_score`
  - `scanner_live_score`
  - `scanner_bet`
  - `scanner_reason`
  - `scanner_signal_type`
  - `scanner_algorithm_basis`
  - `scanner_algorithm_data`

### 6.3. Как AI синхронизируется со scanner

AI не полностью свободен.

Код специально выравнивает ответ модели со scanner:

- для алгоритма 1:
  - confidence AI приближается к `scanner_probability`;
  - если scanner рекомендует ставку, AI не должен слишком сильно уходить в отрицание;
  - в ответ добавляется пометка синхронизации со scanner.
- для алгоритма 2:
  - в ответ добавляется причина scanner;
  - AI анализирует не общий матч, а именно ставку алгоритма 2.
- для алгоритма 3:
  - AI анализирует не матч вообще, а конкретную ставку на выбранную команду;
  - в ответ добавляется синхронизация с правилом scanner и выбранным рынком.

### 6.4. Роль AI в системе

AI сейчас работает как:

- explainability layer;
- second opinion;
- user-facing interpretation of signal.

AI сейчас не является главным движком принятия решения о публикации сигнала.

### 6.5. Что улучшать в AI-слое

- давать модели более структурированные фичи, а не только prose prompt;
- сохранять нормализованный AI-вердикт в отдельные поля;
- измерять, где AI полезно корректирует scanner, а где только повторяет его;
- добавить offline-оценку качества AI-вердиктов по `bet_status`.

## 7. Алгоритм проверки исхода ставки

Это отдельный post-match/post-signal слой в `BetChecker.php`.

Он не ищет новый сигнал, но завершает жизненный цикл анализа.

### 7.1. Для алгоритмов 1 и 2

Ставка считается:

- `won`, если в первом тайме был гол;
- `lost`, если к перерыву счет `0:0`;
- `pending` в остальных случаях.

Фактические условия:

- если `live_ht_hscore + live_ht_ascore > 0` и матч еще в первом тайме или уже в перерыве на 45+ минуте, это `won`;
- если `0:0` и статус `Перерыв`, это `lost`.

### 7.2. Для алгоритма 3

Ставка считается:

- `won`, если выбранная команда забила хотя бы 1 гол;
- `lost`, если матч завершен, а выбранная команда не забила;
- `pending` иначе.

Выбранная сторона берется из `algorithm_payload_json`.

### 7.3. Зачем это важно для новой стратегии

Это уже готовый источник target labels:

- `bet_status = won/lost`

То есть проект уже сам размечает исход части сигналов, и это можно использовать для новой модели.

## 8. Что реально является текущей системой принятия решений

Если упростить:

- Алгоритм 1: полу-эвристическая вероятность гола в 1-м тайме.
- Алгоритм 2: жесткий фильтр на фаворита-хозяев до перерыва.
- Алгоритм 3: жесткий фильтр на ИТБ команды в перерыве по таблице.
- AI: интерпретатор и корректирующий слой, но не основной rule engine.
- BetChecker: разметчик результата ставок.

## 9. Главные ограничения текущей системы

Общие слабые места всех алгоритмов:

- много жестких порогов;
- слабая калибровка под историю win/loss;
- мало учета различий между командами в live;
- нет единой feature-model, где все признаки участвуют вместе;
- мало использования `sgi_json` как богатого сырого источника;
- нет нормального ranking между разными типами сигналов;
- AI не замыкается обратно в dataset обучения.

## 10. Что ИИ должен улучшать

Если ИИ будет предлагать новую стратегию, ему нужно думать в таких направлениях:

1. Не копировать алгоритмы 1/2/3 как есть, а использовать их как baseline.
2. Собрать unified feature set из `matches`, `sgi_json`, `bet_messages`, `ai_analysis_requests`.
3. Разделить признаки на:
   - prematch
   - live
   - historical form
   - H2H
   - table
   - post-signal
4. Проверить, какие из текущих rule thresholds реально работают на истории.
5. Построить score/ranking-модель вместо набора независимых жестких правил.
6. Отдельно оценить:
   - гол в 1-м тайме
   - гол фаворита-хозяев в 1-м тайме
   - ИТБ выбранной команды после перерыва
7. Использовать `bet_status` как уже готовую частичную разметку качества сигналов.

## 11. Практический вывод для новой стратегии

Сейчас проект не имеет одной общей умной модели. Вместо этого он использует три независимых алгоритма:

- один score-based;
- два rule-based;
- и AI как поясняющий слой.

Правильное улучшение системы — это не просто "подкрутить пороги", а:

- собрать полноценный dataset;
- посчитать feature importance;
- понять, какие признаки действительно коррелируют с `won/lost`;
- после этого уже строить новую стратегию, которая либо заменит алгоритмы 1/2/3, либо станет meta-layer над ними.
