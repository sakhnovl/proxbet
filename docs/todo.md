# Техническое задание: добавить в сканер алгоритм 2

## Цель

Добавить в `backend/scanner` второй независимый алгоритм поиска сигнала на гол в первом тайме и обновить формат уведомлений так, чтобы:

- текущий алгоритм явно обозначался как `Алгоритм 1`;
- новый алгоритм явно обозначался как `Алгоритм 2`;
- в Telegram-сообщении и в причине сигнала было понятно, по какому алгоритму найден матч;
- система дедупликации не блокировала отправку сигнала по алгоритму 2, если ранее по этому же матчу уже уходил сигнал по алгоритму 1.

## Текущий контекст

Сейчас сканер работает по одному алгоритму:

- данные собираются через [DataExtractor.php](C:/xampp/htdocs/proxbet/backend/scanner/DataExtractor.php);
- вероятность считается в [ProbabilityCalculator.php](C:/xampp/htdocs/proxbet/backend/scanner/ProbabilityCalculator.php);
- правила допуска к ставке проверяются в [MatchFilter.php](C:/xampp/htdocs/proxbet/backend/scanner/MatchFilter.php);
- итог матча формируется в [Scanner.php](C:/xampp/htdocs/proxbet/backend/scanner/Scanner.php);
- уведомление отправляется из [TelegramNotifier.php](C:/xampp/htdocs/proxbet/backend/scanner/TelegramNotifier.php).

## Требование по алгоритмам

### Алгоритм 1

Ничего не менять в логике отбора текущего алгоритма, кроме явной маркировки:

- все сигналы текущего алгоритма должны считаться сигналами `Алгоритм 1`;
- в `decision.reason` и в Telegram-сообщении должна появиться маркировка `Алгоритм 1`;
- если в проекте есть сохранение сигнала в БД, туда тоже должен попадать идентификатор алгоритма или его текстовая метка.

### Алгоритм 2

Добавить новый алгоритм с отдельным набором условий. Сигнал по алгоритму 2 создается только если одновременно выполнены все условия:

1. Коэффициент на домашнюю команду не больше `1.5`.
2. Коэффициент на тотал больше `2.5` не больше `1.5`.
3. Домашняя команда забивала гол в первом тайме минимум в `3` из последних `5` матчей.
4. В `h2h` голы в первом тайме были минимум в `3` из последних `5` матчей.

## Функциональные требования

### 1. Единая модель результата

В результате анализа матча должна появиться информация об алгоритме, например:

- `algorithm_id`: `1` или `2`;
- `algorithm_name`: `Алгоритм 1` или `Алгоритм 2`;
- `decision.reason`: текст с указанием алгоритма;
- при необходимости `signal_type` или аналогичное поле для дальнейшей аналитики.

Если архитектура позволяет, лучше вернуть не "один матч = один результат", а "один матч = список потенциальных сигналов", потому что теоретически один и тот же матч может удовлетворять сразу двум алгоритмам.

Минимально допустимый вариант:

- оставить текущую структуру, но добавить поле алгоритма;
- если матч подходит под два алгоритма, вернуть два независимых сигнала по одному `match_id`.

### 2. Извлечение данных для алгоритма 2

Нужно определить и использовать существующие поля таблицы `matches` или связанного источника данных для:

- коэффициента на победу домашней команды;
- коэффициента на тотал больше `2.5`;
- статистики домашних голов в первом тайме за последние 5 матчей;
- статистики `h2h` с голами в первом тайме за последние 5 матчей.

Если точные имена колонок отличаются, при реализации нужно:

- найти фактические поля в БД/модели;
- сохранить обратную совместимость;
- не хардкодить новые имена без проверки схемы.

Для алгоритма 2 желательно добавить в `DataExtractor` отдельный метод, например:

- `extractAlgorithmTwoData(array $match): array`

Ожидаемая структура данных:

```php
[
    'home_win_odd' => 1.42,
    'over_25_odd' => 1.47,
    'home_first_half_goals_in_last_5' => 3,
    'h2h_first_half_goals_in_last_5' => 4,
    'has_data' => true,
]
```

### 3. Логика фильтрации для алгоритма 2

Добавить отдельную проверку, не смешивая ее с текущими правилами алгоритма 1.

Рекомендуемый вариант:

- оставить текущий `shouldBet()` как логику алгоритма 1;
- добавить новый метод, например `shouldBetAlgorithmTwo(...)`;
- либо ввести единый роутер правил по `algorithm_id`.

Ожидаемая логика алгоритма 2:

```text
home_win_odd <= 1.5
AND over_25_odd <= 1.5
AND home_first_half_goals_in_last_5 >= 3
AND h2h_first_half_goals_in_last_5 >= 3
```

Для каждой причины отказа нужен понятный текст, например:

- `Алгоритм 2: коэффициент на хозяев выше порога`;
- `Алгоритм 2: коэффициент на ТБ 2.5 выше порога`;
- `Алгоритм 2: домашняя команда редко забивает в 1 тайме`;
- `Алгоритм 2: H2H не подтверждает голы в 1 тайме`.

### 4. Оркестрация в Scanner

В [Scanner.php](C:/xampp/htdocs/proxbet/backend/scanner/Scanner.php) нужно:

- для каждого матча запускать проверку алгоритма 1;
- для каждого матча запускать проверку алгоритма 2;
- формировать один или два сигнала в зависимости от результата;
- не ломать текущую статистику `total`, `analyzed`, `signals`.

Рекомендуемое поведение счетчиков:

- `total`: число матчей, как и сейчас;
- `analyzed`: число матчей, для которых удалось выполнить анализ;
- `signals`: число отправляемых сигналов, а не число матчей.

### 5. Telegram-уведомления

В [TelegramNotifier.php](C:/xampp/htdocs/proxbet/backend/scanner/TelegramNotifier.php) изменить формат сообщения:

- в заголовке или рядом с вероятностью показывать `Алгоритм 1` или `Алгоритм 2`;
- причина внизу должна начинаться с указания алгоритма;
- если алгоритм 2 не использует текущий probability score, это должно быть явно отражено в сообщении.

Пример направления для текста:

```text
СИГНАЛ: ГОЛ В ПЕРВОМ ТАЙМЕ
Алгоритм 2
```

или

```text
Вероятность: 72%
Алгоритм: 1
```

Важно: уведомление не должно вводить в заблуждение. Если алгоритм 2 основан не на вероятностной модели, а на жестких фильтрах, не нужно показывать "вероятность" как главный критерий без пояснения.

### 6. Дедупликация

Сейчас ключ дедупликации строится только от `match_id`. Это нужно изменить.

Новое требование:

- сигнал по `Алгоритму 1` и сигнал по `Алгоритму 2` для одного матча считаются разными;
- повторная отправка того же алгоритма по тому же матчу по-прежнему запрещена.

Рекомендуемый формат ключа:

```text
match_<id>_algorithm_<algorithm_id>
```

Нужно обновить:

- генерацию ключа;
- нормализацию старых ключей;
- совместимость со старым `scanner_state.json`.

## Нефункциональные требования

- Не ломать текущий CLI-режим сканера.
- Сохранить обратную совместимость там, где это возможно.
- Новую логику разделить по ответственностям, без дублирования больших кусков кода.
- Все новые строки причин и уведомлений держать единообразными.
- Не смешивать условия алгоритма 2 с вероятностной моделью алгоритма 1 без явной причины.

## Пограничные случаи

- Если нет данных по коэффициентам, алгоритм 2 должен возвращать отказ с явной причиной.
- Если нет данных по форме хозяев или по `h2h`, алгоритм 2 не должен срабатывать.
- Если матч подходит сразу под 2 алгоритма, должны уйти два отдельных сигнала или быть явно зафиксировано иное правило при реализации.
- Если в БД есть только агрегированное число голов, а не число матчей с голом в 1 тайме, нужно отдельно уточнить соответствие данных и не подменять одно другим.

## Критерии приемки

- В кодовой базе есть отдельная логика алгоритма 2.
- Текущий алгоритм маркируется как `Алгоритм 1`.
- В Telegram-сообщении всегда отображается номер алгоритма.
- Дедупликация учитывает `match_id` и `algorithm_id`.
- Один матч может отправить сигнал по алгоритму 1 и отдельно по алгоритму 2.
- Причины отказа и причины сигнала различаются по алгоритмам.
- При отсутствии нужных данных алгоритм 2 корректно отказывает без падения сканера.
